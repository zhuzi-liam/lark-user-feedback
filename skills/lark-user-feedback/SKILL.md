# lark-user-feedback

用户反馈报告/总结工作流。

**CRITICAL — 开始前 MUST 先使用 `lark-shared` skill 确认认证、权限、`--as` 身份。**
出报告阶段还需配合 `lark-im`（采集）、`lark-doc`（建文档）、`beautiful-feishu-whiteboard`（出画板）。
可选摘要 HTML/PPT 阶段使用 `frontend-slides` 生成 16:9 单文件 HTML；`guizang-social-card-skill` 仅作为 Guizang Swiss / Editorial 视觉风格来源之一，最终仍按 `frontend-slides` 的 deck 规范生成，不直接生成横向 PPT。

## 适用场景

- "看看反馈群里大家对 **仪表盘排行榜组件** 的反馈，出个用户反馈报告"
- "整理一下 **某个反馈群** 里关于 **XX 功能** 的吐槽和需求"
- "把 **XX** 的用户反馈做个需求挖掘 / 优先级排序"

## 启动输入契约

当前版本只接 **飞书群消息**，不接 CRS 工单、内测反馈表、评论区等其他来源。

运行前必须拿到：

| 输入 | 规则 |
|---|---|
| 反馈群 | 用户直接给群名最优；若搜不到或重名，再追问群链接 / chat_id。 |
| 时间范围 | 必填；没给就追问，不默认最近 7 天或 30 天。 |
| 报告主题 / 功能名 | 必填，用于报告标题、关键词扩展和机会点归纳。 |
| 关键词 | 用户可直接给；若没给，先基于主题生成同义关键词草案，让用户确认后再检索。 |
| 可交互 HTML/PPT | 可选。需求澄清最后询问是否需要；用户选择需要时才生成、上传并在报告中落链接。 |

> 例：`分析「多维表格反馈&建议」群，时间 2026-06-01 到 2026-06-09，主题是排行榜组件，关键词：排行榜、榜单、leaderboard。`

## 前置条件

仅支持 **user 身份**（外部群只有 user token 能读）。执行前确保已授权 im + docx + board：

```bash
lark-cli auth status   # 确认 user identity ready，scope 含 im:message search:message docx:document board:whiteboard
```

## 工作流总览

```
{反馈群名} + {时间范围} + {产品/功能主题} + {关键词}
        │
   Step1 采集 ── 群名解析 → 关键词确认 → im +messages-search（多关键词 · 时间窗 · page-all）
        │
   Step2 清洗归类 ─ 四类归档（有效/背景/解法/运营卡片）→ 建内部证据台账(含证据类型·角色·触点·强度)
        │
   Step3 聚合评分 ─ thread 聚合 → 主题聚类 → 5维评分 + 置信度 + 小样本/时效护栏 + 抽贯穿洞察/标主导方 + 口径与格式自检
        │
   Step4 出报告 ─ 开篇洞察 + 6 段(反馈概况/典型用户原声/主题分布/优先级判断/设计机会/下一步验证) + 方法附录 + 多画板 + 原文文字链接 + 可选 HTML/PPT 链接
        │
   Step5 ROI 记录 ── 追加一行到 roi-log.tsv（证明提效）
        │
   Step6 增量更新（可选，见末尾）
```

## Step 1 · 采集

**1.1 校验输入**

- 没有时间范围：先追问，不要自行默认。
- 没有关键词：先给 3~8 个同义关键词草案（核心词、口语变体、英文、报错词），让用户确认后再搜。
- 没有群名 / 群链接 / chat_id：先追问群信息。
- 需求澄清最后询问一次可选增强产物：`是否需要加一个可交互 HTML/PPT？它适合承载更总结性的信息，方便评审/同步时快速浏览。` 选项固定为 `1 需要`、`2 不需要`；两个选项都不要加“推荐”标记，也不要把默认偏好写进选项标题。用户未选择需要时，不生成、不上传、不在报告里预留空模块。

**1.2 群名解析**

用户给群名时，先用 `+chat-search` 找群：

```bash
lark-cli im +chat-search --query "<反馈群名>" --search-types "private,public_joined,external" --page-size 10 --format json --as user
```

- 唯一命中：取 `chat_id` 继续。
- 0 个命中：请用户换群名关键词，或直接发群链接 / chat_id。
- 多个命中：列出候选群名，让用户确认；不要猜。

**1.3 消息检索**

外部群 **不能** 用 `im +chat-messages-list`（会返回空）。必须用 **关键词搜索**，并带上群、时间范围和 user 身份：

```bash
# 对每个确认后的关键词各搜一次，page-all 翻全，落地到文件
for kw in 排行榜 排行 榜单 leaderboard; do
  lark-cli im +messages-search --query "$kw" \
    --chat-id <chat_id> \
    --start "2026-06-01T00:00:00+08:00" \
    --end "2026-06-09T23:59:59+08:00" \
    --as user --page-all --page-limit 40 --format json \
    > raw_${kw}.json
done
```

要点：
- **关键词要扩同义词**：核心词 + 口语变体 + 英文。只搜一个词会漏掉「只描述现象不带词」的反馈，必要时人工补搜（如功能别名、报错关键词）。
- 每条消息含字段：`sender.sender_type`(`user`/`app`)、`sender.name`（可能为空）、`msg_type`(`text`/`post`/`interactive`)、`create_time`、`content`、`message_id`、`thread_id`、**`message_app_link`**（Step4 做跳转按钮要用）。
- 按 `message_id` 去重（不同关键词会搜到同一条）。

## Step 2 · 清洗归类（建内部证据台账）

**不要简单「删掉营销」，而是把每条消息归到四类**，只让「有效反馈」进统计，其余留痕备查：

| 归类 | 判定 | 是否计入频次 |
|---|---|---|
| **有效反馈** | text/post，普通用户在提问 / 报障 / 提需求 / 吐槽 | ✅ 计入 |
| **背景讨论** | 相关但非诉求（如「这组件叫排行榜」纯确认 / 闲聊） | ❌ 不计，可引为上下文 |
| **群内解法** | 用户互相支招、给 workaround（如「最多 10 人，可装插件」） | ❌ 不计为诉求，但**揭示限制**，单列 |
| **运营卡片** | sender=app 的机器人回答，或 user 发的 interactive 营销卡 | ❌ 丢弃 |

运营卡片 title 黑名单（命中即丢）：`排行榜出炉|月度榜单|本期上榜|插件激励|奖励|到账|大赛|作品上榜|优秀作品|报名|直播|预约|重磅更新|模板|应用模式|开箱即用|获取同款`。

**每条「有效反馈」和「群内解法」都进内部证据台账**，字段：`ID(message_id) · 日期 · 用户友好名称 · 来源群 · 原话 · 跳转链接 · 证据类型 · 角色 · 触点 · 强度`。读者版报告不要展示 `open_id / user_id`。日期粒度到 `YYYY-MM-DD` 即可，除非用户明确要求审计日志，不展示分钟级时间。

用户展示规则：优先展示消息中可取得的 `sender.name` / 昵称；取不到时降级为「外部用户 · 日期」。取不到昵称不阻塞报告生成。

**证据类型（关键，区分一手 / 二手）**：
- `用户原话`（一手 · 真实诉求）/ `群内解法`（二手 · 别人支招）/ `官方说明`（app / 官方）/ `代理转述`（你的归纳）。
- 规格类结论（如「10 人上限」）若来源是 `群内解法`，必须标 `待产品确认`，**不能写成既定事实**。

**角色 / 触点 / 强度（关键三维 —— 设计报告的可操作性全靠它们）**，每条都标，靠原话推断：
- **角色 = 谁在反馈**：`制作者`（搭这个功能的人）/ `查看者`（领导等看结果的人）/ `客户`（外部，通常经转述）/ 其它。**这是设计报告最该有、却最容易丢的一刀** —— 不同角色的痛和解法完全不同；同一现象由制作者 vs 查看者提出，价值天差地别。⚠️ 留意「沉默大多数」：查看者/客户很少进反馈群，他们的反馈少 ≠ 问题小，要在「下一步验证」里安排主动验证。
- **被影响对象不要做成固定列**：只有当反馈角色、使用者、被影响者可能混淆时，才在 `角色` 或证据摘要里内联标注（如「制作者（转述领导查看场景）」）。无歧义时不要额外加「被影响对象」列，避免表格变重。
- **触点 = 在用户旅程的哪一步出问题**：`配置态`（搭建/设置时）/ `查看态`（看结果时）/ `分享态`（分享/对外后）/ `入口`。决定「设计该在哪一步改」。
- **强度 = 诉求的紧迫度**：🔴 对外升级（客户级事故）· 🟠 明确受阻（用不了/严重影响）· 🟡 需求或疑问（想要/在问）。防止把 15 条等权看待，让真痛点浮上来。

> 经验：真实反馈 = 少字 text/post、发送者分散；营销 = interactive 卡、来自 1~2 个重复账号。拿不准逐条人工判，宁可多看几条。

## Step 3 · 聚合、评分与置信度

**3.1 去重 + 线程聚合**：先按 `message_id` 去重；再按 `thread_id`（+ 同一用户、±同日）聚合 —— 同一条讨论被拆成多条不能重复计数。统计时同时给出 `原始条数` 和 `聚合后独立诉求数`。

**3.2 主题聚类**：把有效反馈合并成 4~6 个主题，**每个主题用简短描述性名字**（如「数据正确性与刷新」「可见性/端与链接」）。**不要用 T1/T2 代号**（读者会误以为是优先级）；优先级一律用 P0/P1/P2 单独表达。每个主题记 `反馈条数(聚合后) · 反馈人数 · 代表原话 · 证据类型分布`。
> 反馈人数 = 去重后的反馈人数量。它能区分「1 个人反复追问」和「多人都遇到」。

- **根因异质护栏**：按症状聚类（如「看不见」捆了旧版 / 移动端 / Mac / 分享链接四个可能不同根因，或「数不对」捆了不刷新 / 不算总和 / 显示全相同等多个失败模式）时，主题依据栏须注明『含 N 个可能不同根因 / 失败模式』，该主题的频次不得用于断言某个单一问题的严重度。**这条对每个主题一视同仁**——别只给证据旧的主题标异质、却放过证据多的主题（双标）。
- **弱信号不立主题**：聚合后仅 1 条 / 1 个独立用户的信号并入「其他信号」区，不单独成主题、不参与 P 级排序（仍进证据台账留痕）。
- **主题命名唯一**：每个主题定稿一个名字（如 `可见性/端与链接`），正文、表格、画板全文逐字复用，不许出现同主题多种叫法。优先级（P0/P1/P2）与主题是两套独立信息，分列呈现，别用同一个前缀混在一起。
- **不在「主题」列里重复「主题」二字**：当表格/画板某列表头已是「主题」时，格子直接写名字（`可见性/端与链接`），**不要写成「主题2 可见性/端与链接」**——表头已表明这是主题，再加前缀就是同词说两遍。需要在正文/画板说明里指代某主题时，用引号包名字（如「可见性/端与链接」），不依赖编号。

**3.3 优先级评分（轻量 rubric，别造假精度）**：每个主题对下列维度打 `高/中/低`，给 P 级时附**一句话依据**：

| 维度 | 问什么 |
|---|---|
| 频次 | 聚合后多少条 / 多少独立用户 |
| 影响面 | 影响多少类用户、是否核心场景 |
| 阻断程度 | 「完全用不了」还是「不够好用」 |
| 替代成本 | 有没有 workaround、绕道贵不贵 |
| 复现确定性 | 是否多人、多端、多时点复现 |

**3.4 置信度 + 小样本/时效护栏（最重要）**：
- 每个主题标 `置信度 高/中/低` + 理由。
- **样本护栏**：单主题聚合后 `反馈人数 ≤ 2`，或全量有效反馈 `< 8` 时 —— 优先级一律标「**暂定**」，置信度上限「中」，**禁止断言「最痛 / 必做」**，结论改为「方向性信号，待验证」。报告须显式写明：这是定性方向，非统计结论。
- **时效护栏**：时间窗超过 6 个月时，主题分布必须呈现各主题的时间分布（如「近半年 N/总数」）；距今超过 1 年的版本 / 端 / 平台类证据一律标「**需复核现状**」（这类问题很可能已被产品迭代修复），不得以全权重支撑 P0/P1。某主题证据**全部**距今超过 6 个月时，P 级须标「**待复核**」，并把『先复核现状』写进该主题的验证步骤。

**3.4b 从「整理」升到「洞察」（决定报告值不值钱的一步）**：分完主题别就停在「4 个并列的桶」，再往上抽一层：

- **找一条贯穿性洞察（through-line）**：把零散主题串成一个能让人记住的判断。常见形态——**链路/层级**（如「数据对→看得到→看得清→看得懂」的信任链，断在哪一环）、**人群分裂**（制作者 vs 查看者要的不是一回事）、**阶段集中**（问题都卡在配置态）。一份报告**至少给一条**这样的洞察，放在最前面（开篇结论 + 一个「核心洞察」callout/画板）。没有它，报告只是台账的美化版。
- **痛点要写到「代价」**：每个重点主题不止写现象（「数值全 0」），要写**后果/代价**（「对外汇报时数字明显错→当场失信→怀疑整个仪表盘→弃用」）。代价才是优先级的真正依据，也是打动决策者的钩子。
- **标「主导方」——这是 *设计* 报告的灵魂**：每个机会标清是 `设计主导`（信息架构/布局/交互，设计能拍板）/ `设计兜底`（底层是工程 bug 或产品能力边界，设计只能做异常态、说明、引导）/ `产品·设计共担`。**让设计师一眼知道「哪个该我接、我能主导多少」**，别把一堆工程 bug 包装成「设计机会」。
- **角色 / 触点要在主题层做聚合**：每个主题标「主要角色 · 主要触点」，并留意人群是否分裂、问题是否集中在某个触点——这往往就是上面那条贯穿洞察的来源。

**3.5 口径自检（出报告前必过）**：`各主题反馈数之和（含「其他信号」）` 必须等于 `台账里有效反馈条数`；对不上就回查，不允许「有效 9 / 主题合计 6」这种矛盾。背景讨论、群内解法不计入主题频次。

**3.6 产出格式核对（出报告前必过）**：
1. 证据表含「**证据类型**」列（用户原话 / 群内解法 / 官方说明 / 代理转述）——规范里写了但生成时最容易漏的一列；
2. 主题名在正文、表格、所有画板中**逐字一致**；
3. 画板上的数字、P 级、置信度与正文表格**同口径**；
4. 时间窗 >6 个月时，时间分布列与「需复核现状」标记已落到主题分布和证据表；
5. 证据表含「**角色**」与「**强度**」信号，主题分布表含「**主要角色 · 触点**」；
6. 有**至少一条贯穿性洞察**（见 3.4b）写在开篇，且每个重点主题标了「**主导方**」和「**代价**」；
7. 「下一步验证」是可认领的表（含 Owner / 目标日期空列），且包含一项「**主动验证沉默人群**」（查看者/客户）。
8. **画板去重检查**：逐块检查画板标题、视觉编码和表达目的；任意两块画板不能复用同一份数据和同一种表达。尤其「核心洞察」必须表达贯穿判断/链路/人群分裂/阶段卡点，「主题分布」才表达频次/人数/时间分布；两者不得都做主题频次速览。
9. 若用户选择生成 HTML/PPT：报告一句话结论下方已放云空间链接；HTML/PPT 与正文、画板没有复刻同一份主题频次图；本地 HTML 已做桌面 + 手机视口验收。

## Step 4 · 出报告（读者版文档 + 多画板 + 原文按钮）

用 `lark-doc` 建 docx，正文承载结论文字，可视化用 `beautiful-feishu-whiteboard` 出板。文档按 `lark-cli` 默认行为创建到操作者自己的飞书云端空间，不配置固定知识库或固定文件夹。**出板前必须读取 `beautiful-feishu-whiteboard` 的 `SKILL.md / RULES.md / CATALOG.md`。画板默认使用 `Long Table`（`templates/long-table/`）视觉风格；若内容形态明显不适合长表格表达，再退到 Restrained，其次 Balanced；不要用 Bold 风格，除非用户明确指定。**

**画板不是正文表格的复刻，也不能彼此重复**：每块画板必须有唯一表达目的，并至少包含一种正文表格给不了的视觉编码——频次条形（可叠加反馈人数第二条）、2×2 优先级矩阵、流程箭头、时间轴/时间分布。和正文表格内容完全相同的纯表格画板，要么升级编码，要么删掉只留正文表格。不同章节不能复用同一个 SVG / 同一组主题频次图；如果两块画板标题不同但主体数据、布局和结论相同，必须改成不同分析视角或删掉一块。画布尺寸裁剪到内容边界，避免缩略图大片空白。

报告结构 = **开篇洞察 + 6 段 + 附录**。读者版优先呈现结论、证据和行动，不暴露低价值采集过程细节。核心分析章节尽量用画板承载，正文补充图中放不下的证据与判断。

0. **开篇结论 + 核心洞察** —— 「一句话结论」必须先说人话：主要问题是什么、设计师先做什么、哪些需要产品/工程复核。不要把分析框架、隐喻、优先级、风险提示和角色洞察都塞进一句里；抽象框架（如信任链、制作者 vs 查看者分裂）放到后面的「核心洞察」callout 或画板展开。句式尽量直给，非必要不用「不是……而是……」。
   - 若用户选择生成可交互 HTML/PPT：链接固定放在一句话结论下面，文案用 `可交互摘要 PPT：<云空间链接>`。这个入口只承载更总结性的信息，不替代正文结论、画板和证据表。
1. **反馈概况** —— 一句话结论 + 群范围 + 日期级时间范围 + 有效反馈数 + 优先判断 + **局限/分母**（如「15 条/近两年/5 群是定性线索，非使用规模」，防止读者误推「多少人受影响」）。不要写 `00:00:00 / 23:59:59`、关键词原始命中数、去重后消息数、page 数、剔除规则明细。
2. **典型用户原声** —— 保留这个章节名，不要改成「问题证据台账」。按主题拆成若干张小表，并按主题优先级/反馈量排序；每张表包含：ID · **角色** · 日期 · **功能点位** · 触点 · **用户原文**（前缀**强度图标** 🔴/🟠/🟡）· 用户期望/影响 · **证据类型** · 原文。读者可见 ID 必须按最终展示顺序从 `E01` 连续编号，不沿用采集顺序或原主题顺序；设计机会、验证项、附录、画板里的证据引用都必须同步使用这套展示 ID。不要只写摘要，必须贴用户反馈原文；必要压缩时只清理换行和无意义空白，不改写成代理总结。功能点位按产品点位标注（如 Table Agent / AI 侧边栏 / 单元格 AI / 数据表工具栏 / 数据表智能筛选），在表格中用 `<code><span text-color="...">...</span></code>` 或等价标签样式呈现，不用正文样式；不同功能点位必须使用不同字色，且同一功能点位全文保持同一字色。不要单独放「来源群」列，把来源群名放进「原文」列，做成文字超链接。不要展示 `open_id / user_id`；昵称只有在可读且有价值时展示。原话里能推断出**角色/场景**（如「领导在移动端看」）的信号必须保留。距今超 1 年的版本/端类证据在日期旁加 ⚠️，表下脚注解释 ⚠️ / 证据类型 / 角色 / 强度 四个标记。
3. **主题分布** —— 画板（频次条形 + 时间分布圆点）+ 表格：每主题 `反馈条数 · 反馈人数 · 主要角色·触点 · 时间分布 · 置信度 · 代表卡点与依据`。时间窗 >6 个月必须带时间分布；依据栏注明根因/失败模式是否异质。不要写成 `反馈/用户` 这种斜杠表头。
4. **优先级判断** —— 表格：5 维 rubric + P 级 + **置信度** + 「暂定 / 待复核」标记。小样本不写「最痛」；证据全旧的主题不给实 P0。配 callout 点出异质、时效、二手等口径风险。
5. **设计机会** —— 画板（任务→卡点→机会→验证流程 + **主导方徽章** + 支撑证据 ID）+ 表格，对 **Top 2~3 主题**写成：`主导方 · 支撑证据 · 用户任务 · 当前卡点→代价 · 设计机会 · 验证方式`。**每行标主导方（设计主导/设计兜底/共担），卡点写到代价**；其余主题列简版。所有 `E##` 支撑证据 ID 都必须可点击；区间引用（如 `E20-E29`）要展开成逐个可点击 ID，不能留下不可点击文本；多个证据 ID 之间必须使用可见分隔符（优先顿号 `、`），不要只用空格，防止飞书渲染后链接粘连。
6. **下一步验证** —— 表格：待验证项 · 要确认什么 · 找谁/哪个源 · 产出 · **Owner · 目标日期**（后两列留空待认领）。必须含一项「**主动验证沉默人群**」（查看者/客户）。聚焦可执行，不堆模板。
7. **附录 · 方法说明**（短，放文末，可用灰底 callout）—— 检索关键词、raw→valid 漏斗数、清洗与聚合口径各一句话、生成日期与操作者、已知局限（如「仅关键词检索，可能漏掉纯截图反馈」）。过程细节放这里而不是正文（与 RULES 9 呼应），让同事能复核、能复跑。**没有真实数字就写定性描述，禁止编造漏斗数。**

> **设计机会 vs 功能建议**：用户是设计师，输出别停在「加个升级入口 / 提高人数上限」这种功能清单；要回答「用户想完成什么任务、卡在哪、设计上有什么机会、怎么验证」。功能点是结论的副产品。

**跳转原文链接（强烈建议保留，这是和普通 AI 摘要的最大差异）：**
每条原话挂一个文字超链接，点了直接跳回群内对应话题：
```xml
<a href="转义后的 message_app_link">Base Feedback 2群</a>
```
- `message_app_link` 直接取自 Step1 每条消息；形如 `https://applink.feishu.cn/client/thread/open?...&open_thread_id=omt_...`。
- `<a href>` 可放进 `<p>` 或表格 `<td>`。**href 里的 `&` 必须转义成 `&amp;`**，否则链接断。
- 在「典型用户原声」表里，链接文案默认用具体来源群名（如 `Base Feedback 2群`），不再单独放「来源群」列。
- 如果某个场景不需要展示来源群，链接文案才退回短标签「原文」。
- 典型原声表的 `ID` 也要做成链接；点击 `E##` 与点击「原文」都能跳到对应原文证据。
- `用户原文` 和 `用户期望/影响` 两列要给更宽的列宽；其他列维持紧凑，避免正文挤压到不可读。

文档生成后，用 `docs +media-download --type whiteboard --token <block_token>` 把每块板下载缩略图**肉眼核对**（尤其检查深底标签是否可读、文字是否截断、是否空白）。避免“只有一张速览图，后面全是表格”。

### 可选 HTML/PPT 摘要模块

仅当用户在需求澄清中选择“需要”时执行；不要默认生成。

1. **风格择优**：访谈最后给 3 个沉稳、专业、适合反馈报告的视觉预览。预览来源不设固定数量配比，可以全来自 `frontend-slides`，也可以包含 Guizang-inspired；按 `报告内容契合度 > 可读性 > 专业感 > 差异度 > 实现稳定性` 排序推荐。推荐方向：Long Table / Signal / Cartesian / Guizang Swiss / Guizang Editorial；避免强营销、过度装饰、泛 AI 风格。
2. **生成规范**：使用 `frontend-slides` 生成单文件 HTML，固定 1920×1080 stage，按 16:9 等比缩放；完整遵守 `frontend-slides` 的 `viewport-base.css`、HTML 结构、动画与验收规则。
3. **Guizang 边界**：`guizang-social-card-skill` 只作为视觉语言来源，不直接生成横向 PPT，不输出 3:4 社交卡，除非用户另行明确要求传播图/封面/卡片。
4. **内容边界**：HTML/PPT 优先承载总览、关键洞察、场景矩阵、行动建议；不得复刻报告正文表格或画板里的同一份主题频次图。
5. **验收与上传**：生成后必须用桌面 + 手机视口检查无空白页、无溢出、无重叠；通过后上传飞书云空间，并把云空间链接插入到报告一句话结论下面。原报告文档保留，不用 HTML/PPT 替代。

### 已有报告迭代 / 评论处理

如果用户说“我在报告里留了评论，帮我改一下”：
- 先用 `lark-drive` 读取未解决评论：`drive file.comments list --params '{"file_token":"<doc_token>","file_type":"docx","is_solved":false}' --as user --format json`。
- 逐条把评论转成文档改动，不要只口头解释。
- 修改后做关键词核对，确认评论指出的旧文案、用户 ID、过细时间、低价值统计没有残留。
- 把可复用规则沉淀回本 skill。

## Step 5 · ROI 记录（每次必做）

> **为什么要做**：判断一个 skill 值不值得继续投入，靠的是「**省了多少时间 + 质量是否稳定 + 被多少人复用**」，而不是「产出了几份文档」。不记录，就只能说「我做了个东西」，无法证明提效，也没法决定要不要扩大。所以每跑一次，追加一行到 [`roi-log.tsv`](roi-log.tsv)：

| 字段 | 含义 | 例 |
|---|---|---|
| date | 跑的日期 | 2026-06-05 |
| topic | 报告主题 | 仪表盘排行榜组件 |
| source | 数据源 | Base Feedback 群 |
| raw / valid | 捞到条数 / 有效反馈数 | 46 / 9 |
| manual_min | 人工做同样一份**估计**耗时(分钟) | 180 |
| skill_min | 本次 skill 实际耗时(分钟) | 25 |
| rework | 产出后需返工的程度(无/小/大) | 小 |
| reuse | 复用人/场景 | 设计+PM |
| note | 备注(踩坑/改进) | 画板深底标签需修对比度 |

```bash
# 追加一行（TSV，制表符分隔）
printf '%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\n' \
  "2026-06-05" "仪表盘排行榜组件" "Base Feedback群" "46/9" "180" "25" "小" "设计+PM" "深底标签对比度" \
  >> "$HOME/.codex/skills/lark-user-feedback/roi-log.tsv"
```
跑完后用一句话向用户汇报：本次省了约 `manual_min - skill_min` 分钟，有效反馈 N 条，返工程度 X。

## Step 6 · 增量更新（可选 / 预留）

反馈持续流入，避免每次全量重跑：
- 在 `roi-log.tsv` 旁记录上次跑的 **最新 message create_time 水位**。
- 下次采集时对结果按 `create_time > 上次水位` 过滤，只分析新增，再 `docs +update` 追加到既有报告或单出「增量篇」。
- MVP 阶段可先不做；数据量大或需要周期性复盘时再开启。

## RULES（踩过的坑，务必遵守）

1. **飞书画板会把所有 `<text>` 强制渲染成深色** —— 任何「深底 + 浅字」都会变成「深底 + 看不见的深字」。**绝不能用浅色文字压深色填充**。优先级靠「填充深浅 + 描边粗细 + 位置」编码，标签一律深色文字放在浅底上。详见模板文件。
2. **外部群**：`im chats.get` 可能报 232033（无权限），用户姓名可能解析不到 → 优先用消息里的 `sender.name`，取不到就降级为「外部用户 · 日期」；消息只能 `im +messages-search` 按词搜，不能列全量。
3. **运营卡片必剔 + 四类归档**（Step2），否则结论被营销噪音污染。
4. **小样本不强判**：样本薄（独立用户 ≤2 / 有效 <8，见 3.4）时优先级标「暂定」、**不写「最痛 / 必做」**，报告注明「方向性、非统计」。报告越漂亮越要守这条——别让精致排版给薄数据背书。
5. **口径自检**：主题频次之和 = 台账有效条数，对不上不准发（见 3.5）。
6. **一手 / 二手分清**：群内解法 / 答疑 ≠ 用户诉求；规格类结论标 `待产品确认`。
7. 报告**结论既进画板也进文字/callout** —— 画板是「一图速览」增强，不是唯一载体。
8. 出板后**必下载缩略图肉眼验收**，别盲发。
9. 读者版报告默认不写采集过程细节：关键词原始命中数、去重后消息数、page 数、`message_id` 去重方式、运营卡片剔除明细等放内部台账 / ROI / 附录；正文只保留影响读者判断的信息。
10. 时间写到日期即可（`YYYY-MM-DD`）；除非用户明确要审计日志，不写分钟级时间或 `00:00:00 / 23:59:59`。
11. 证据表不展示 `open_id / user_id`。保留日期、用户原文、功能点位、证据类型和「原文」文字链接；来源群不单独成列，放进「原文」链接文案里。
12. 用户反馈报告要多图形化呈现：主题分布、优先级矩阵、机会路径、验证矩阵优先用 `beautiful-feishu-whiteboard`；每块图都要经过渲染检查和肉眼验收。
13. 画板风格选择默认优先级：Long Table > Restrained > Balanced；避免 Bold。只有用户明确要求强视觉冲击、海报化或品牌活动风格时，才考虑 Bold。
14. **时效护栏**（见 3.4）：时间窗 >6 个月必须给主题时间分布；距今 >1 年的版本/端类证据标「需复核现状」；证据全旧的主题 P 级标「待复核」。两年时间窗里 2024 年的"手机端不显示"很可能早修复了——别让陈年证据撑起今天的 P0。
15. **产出格式核对**（见 3.6）：证据类型列存在、主题名全文逐字一致、画板与正文数字同口径、画板之间不重复。规范写了 ≠ 生成时做到了，发布前逐项勾。
16. **画板禁复刻和互相复用**：同一份矩阵在速览板、主题分布板、正文表格出现三遍是冗余不是丰富；同一份主题频次图不能同时当「核心洞察」和「主题分布」。画板必须带正文表格给不了的视觉编码（条形/矩阵/流程/时间轴），且每块画板的分析问题不同：核心洞察讲贯穿判断，主题分布讲数量结构，优先级讲取舍，设计机会讲行动路径。
17. **角色/场景信号不洗掉**：原话里出现「领导看」「移动端汇报」「分享给客户」这类角色与场景线索时，必须保留进证据摘要——它决定设计机会值不值钱。
18. **「典型用户原声」按主题拆表**：章节名固定保留为「典型用户原声」。不要生成一张超宽大表；按主题拆成多张小表，主题顺序跟「主题分布/优先级判断」一致。无歧义时不加「被影响对象」列；有歧义时在角色或证据摘要内联说明。
19. **原文列合并来源群**：「典型用户原声」不要单独放「来源群」列。把具体群名放到「原文」列里，并用这个群名作为文字超链接文案；不要用按钮。只有不需要展示群名时，链接文案才退回「原文」。
20. **证据 ID 按展示顺序编号**：读者看到的 ID 是阅读索引，不是数据库主键。定稿前按「典型用户原声」从上到下重排为 `E01、E02、E03...`；如果主题顺序调整，必须重新编号，并同步更新所有正文引用、设计机会表、下一步验证、附录和画板。禁止出现第一个证据从 E05 开始的情况。
20b. **证据 ID 必须可点击且有分隔**：「典型用户原声」里的 `E##`、场景矩阵/设计机会/验证项里的支撑证据都要链接到对应原文证据。引用区间必须展开为逐个可点击 ID；不能留下 `E20-E29` 这种不可点击范围文本。多个 ID 之间必须用顿号等可见分隔符，不要只用空格；发布前回读检查不能出现 `E03E05` 这类粘连。
21. **读者面措辞要直白，少行话**：章节标题、表头、画板标签都给"没参与反馈整理的人"也能秒懂的词。具体：
    - 别用易混淆行话——**「断点」→「卡点」**（断点会被读成 CSS/调试断点）；**「主题地图」→「主题分布 / 概况」**（地图太抽象）。
    - 表头**不中英混排**——`Top 主题 → 重点主题`；缩略词**写全**——`反馈/用户 → 反馈条数 + 反馈人数`。
    - 但**评分量表维度**（频次 / 影响面 / 阻断程度 / 替代成本 / 复现确定性）是刻意精确的定义词，保持原样，别为"通俗"牺牲口径。
    - 判断法：一个词如果需要你脑补它指什么，读者也会——换成不用脑补的说法。
    - 「一句话结论」优先用行动语言，不优先用隐喻：`主要问题是 X；设计师先处理 Y；产品/工程复核 Z`。
    - 避免非必要的「不是……而是……」句式；如果要做对比，优先改成正向判断。
22. **从整理升到洞察**（见 3.4b）：报告别停在「N 个并列主题」。必须有①一条贯穿洞察写在开篇、②每个重点主题标「主导方」（设计主导 / 设计兜底 / 共担）、③痛点写到「代价」、④角色/触点维度。没有这四样，它只是台账美化版，帮不到设计师发现机会。
23. **警惕沉默大多数**：反馈数量是「谁在反馈群里说话」的产物，不是「谁有问题」的全貌。查看者、外部客户很少进群，他们的反馈少 ≠ 问题小。报告要点名这一偏差，并在「下一步验证」安排主动验证，别让反馈数量直接当用户规模。
24. **典型原声表可读性**：按主题拆表后仍要保证列宽。`用户原文` 与 `用户期望/影响` 两列必须明显更宽；`功能点位` 用 `<code>` / 标签样式，不能混成正文；不同功能点位使用不同字色，同一功能点位全文保持同一字色；`用户原文` 必须来自真实反馈内容，不能只放代理摘要。
25. **可交互 HTML/PPT 是可选增强，不是默认产物**：只有用户选择需要时才生成、上传、落链；链接固定放在一句话结论下面。`frontend-slides` 负责 16:9 单文件 HTML/PPT 生成与验收，`guizang-social-card-skill` 只提供可选视觉风格参考；不要让 Guizang 直接生成横向 PPT，也不要写死两个 skill 的风格候选数量比例。HTML/PPT 要承载总结性信息，避免和报告画板重复表达同一张主题频次图。
